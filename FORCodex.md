# FORCodex

## 이 프로젝트를 한 문장으로 말하면
이 앱은 "내 프로젝트들을 한 화면에서 정리해서 보는 주소록"에 가깝습니다.

예전에는 서버 상태를 감시하는 보드에 더 가까웠지만, 지금은 훨씬 단순합니다.  
핵심은 프로젝트별로 중요한 정보를 저장하고, 나중에 빠르게 다시 찾는 것입니다.

비유하면 이런 느낌입니다.

- 예전 방향: 건물 경비실 모니터
- 지금 방향: 잘 정리된 프로젝트 카드 보관함

둘 다 쓸모는 있지만, 목적이 다릅니다.  
이번 프로젝트는 "복잡한 자동 감시"보다 "쉽게 저장하고 다시 보는 것"에 더 집중하고 있습니다.

## 지금 저장하는 정보
현재 각 프로젝트는 아래 정보를 가집니다.

- `Project Name`: 프로젝트 이름
- `Description`: 프로젝트 설명
- `Output Format`: 결과물이 어떤 형태인지
- `Server Location`: 서버가 어디에 있는지
- `URL`: 서비스 주소
- `URL(Admin)`: 관리자 페이지 주소
- `Tech Stack`: 사용 기술 목록

여기서 이번 변경의 핵심은 `URL(Admin)`입니다.  
서비스 주소와 관리자 주소는 실제 운영에서 자주 다릅니다. 그래서 둘을 따로 저장하는 게 맞습니다.

예를 들면:

- `URL`: 사용자가 보는 메인 서비스
- `URL(Admin)`: 운영자가 로그인해서 관리하는 백오피스

이 둘을 한 칸에 억지로 넣으면 나중에 헷갈립니다.  
좋은 엔지니어는 "지금은 비슷해 보여도 역할이 다르면 분리"합니다.

## 구조를 아주 쉽게 설명하면
이 프로젝트는 크게 두 덩어리입니다.

### 1. `client/`
브라우저에 보이는 화면입니다.

중요 파일:

- `client/src/pages/ProjectsPage.jsx`
  메인 목록 화면입니다. 테이블 헤더와 프로젝트 목록을 그립니다.
- `client/src/components/AddEditModal.jsx`
  프로젝트 추가/수정 팝업입니다.
- `client/src/components/ProjectRow.jsx`
  테이블의 한 줄을 그립니다.
- `client/src/components/ProjectAccordion.jsx`
  행을 펼쳤을 때 보이는 상세 카드입니다.

쉽게 말해 `client/`는 손님이 직접 보는 매장입니다.

### 2. `server/`
데이터를 저장하고 꺼내주는 쪽입니다.

중요 파일:

- `server/app.js`
  서버 시작점입니다.
- `server/routes/projects.js`
  프로젝트 조회, 추가, 수정, 삭제를 처리합니다.
- `server/lib/dataStore.js`
  JSON 파일을 읽고 쓰기 전에 데이터를 정리합니다.

쉽게 말해 `server/`는 창고와 계산대 역할을 같이 합니다.

## 데이터는 어디에 저장되나
이 프로젝트는 복잡한 데이터베이스 대신 JSON 파일을 사용합니다.

왜 이렇게 했을까?

- 지금 규모에는 더 단순합니다.
- 설치와 운영이 쉽습니다.
- 비개발자도 데이터 구조를 이해하기 쉽습니다.

이건 "작은 노트에 정리해두기"에 가깝습니다.  
아직은 대형 창고(DB)까지 갈 필요가 없는 단계라고 판단한 것입니다.

## 화면에서 저장까지 어떻게 연결되나
프로젝트를 추가할 때 흐름은 이렇습니다.

1. 사용자가 `AddEditModal`에 값을 입력합니다.
2. 브라우저가 `/api/projects`로 저장 요청을 보냅니다.
3. 서버의 `routes/projects.js`가 요청을 받습니다.
4. `dataStore.js`가 데이터를 정리합니다.
5. JSON 파일에 저장됩니다.
6. 다시 목록을 불러와 화면에 보여줍니다.

수정도 거의 같습니다.

1. 사용자가 수정 버튼을 누릅니다.
2. 모달에 기존 값이 채워집니다.
3. 저장하면 `/api/projects/:id`로 수정 요청이 갑니다.
4. 서버가 같은 프로젝트를 찾아 값을 덮어씁니다.
5. 목록이 다시 로드됩니다.

여기서 중요한 포인트는 하나입니다.

화면에 입력칸만 추가하면 끝이 아닙니다.  
서버도 그 값을 받아야 하고, 저장 정리 코드도 그 값을 버리지 않아야 합니다.

이번 작업에서 바로 그 부분을 같이 고쳤습니다.

## 이번 변경에서 실제로 바뀐 것
이번 요청은 보기보다 범위가 조금 넓었습니다.

겉으로는:

- 테이블에 `URL(Admin)` 열 추가
- 추가/수정 폼에 `URL(Admin)` 입력칸 추가

하지만 실제로는 저장 흐름까지 함께 바뀌어야 했습니다.

그래서 아래가 같이 수정됐습니다.

- `ProjectsPage.jsx`
  테이블 헤더에 `URL(Admin)` 추가
- `ProjectRow.jsx`
  각 행에 관리자 URL 표시
- `AddEditModal.jsx`
  입력 폼에 `URL(Admin)` 추가
- `projects.js`
  API가 `adminUrl`을 저장하도록 변경
- `dataStore.js`
  정리 과정에서 `adminUrl`이 사라지지 않도록 변경
- `ProjectAccordion.jsx`
  열 수가 늘어난 만큼 상세 행 너비를 맞추고, 상세 카드에도 관리자 URL 표시

이 마지막 항목이 좋은 실무 포인트입니다.

테이블 열을 하나 늘리면 끝처럼 보여도,  
펼침 행의 `colSpan`도 같이 맞춰야 화면이 안 깨집니다.

즉, 좋은 엔지니어는 "보이는 변경"만이 아니라  
"그 변경 때문에 조용히 같이 깨질 수 있는 부분"도 같이 봅니다.

## 이번에 배울 수 있는 실무 감각
### 1. 같은 "URL"이어도 역할이 다르면 필드를 분리한다
메인 서비스 주소와 관리자 주소는 목적이 다릅니다.

- 보는 사람도 다르고
- 접근 권한도 다르고
- 나중에 쓰는 맥락도 다릅니다

그래서 이름을 분리한 것은 단순한 취향이 아니라, 데이터 설계의 문제입니다.

### 2. 화면 변경은 종종 저장 구조 변경이다
초보자는 "입력칸만 넣으면 되지 않나?"라고 생각하기 쉽습니다.

하지만 실제 앱에서는:

- 화면
- API
- 저장 포맷
- 상세 보기

가 다 연결되어 있습니다.

한 군데만 바꾸면 저장이 안 되거나, 새로고침 후 값이 사라지거나, 화면 정렬이 깨질 수 있습니다.

### 3. 작은 버그는 대개 연결 부위에서 생긴다
이번 작업에서 조용히 같이 챙겨야 했던 부분이 `ProjectAccordion.jsx`의 `colSpan`이었습니다.

이런 건 "기능 자체"보다 "연결면"에서 생기는 버그입니다.

실무에서는 이런 연결면 체크가 중요합니다.

- 컬럼 수가 바뀌면 상세 행 폭도 맞는가?
- 새 필드가 수정 모드에서 기존 값으로 채워지는가?
- 저장 후 새로고침해도 남아 있는가?

이 질문을 자동으로 떠올릴수록 엔지니어링 감각이 좋아집니다.

## 이번에 부딪힌 문제와 해결
### 문제 1. 새 필드를 화면에만 넣으면 저장이 안 된다
`AddEditModal.jsx`에 입력칸만 추가하면 브라우저에서는 값을 적을 수 있습니다.

하지만 서버가 그 값을 모르면 저장 시 버려집니다.

해결:

- `server/routes/projects.js`에 `adminUrl` 추가
- `server/lib/dataStore.js`에 `adminUrl` 유지 로직 추가

### 문제 2. 테이블 열 추가 후 상세 영역 너비가 어긋날 수 있다
Projects 테이블은 열 수가 늘었는데, 펼침 상세 영역은 예전 열 수 기준이면 레이아웃이 어색해질 수 있습니다.

해결:

- `ProjectAccordion.jsx`의 `colSpan`을 `7`로 조정

### 문제 3. 문서가 코드보다 뒤처질 수 있다
비개발자 입장에서는 코드보다 문서가 먼저 이해의 기준이 됩니다.

문서가 오래된 구조를 설명하면, 실제보다 더 혼란스러워집니다.

해결:

- `PROGRESS.MD`에 이번 단계 기록
- 이 문서(`FORCodex.md`)를 현재 구조 기준으로 갱신

## 왜 이 선택이 "가장 단순한 안정 해법"인가
더 복잡하게도 만들 수 있었습니다.

예를 들면:

- URL 타입을 여러 개 동적으로 추가하게 만들기
- `public/admin/internal/docs`처럼 URL 종류를 배열로 관리하기
- 별도 URL 관리 컴포넌트를 새로 만들기

하지만 지금 단계에서는 과합니다.

지금 필요한 건 딱 하나였습니다.

- 관리자 URL을 따로 저장하고 보여주기

그래서 가장 안정적인 선택은 기존 구조를 유지한 채 `adminUrl` 한 필드만 추가하는 것입니다.

이게 바로 "필요한 만큼만 바꾸는" 좋은 수정입니다.

## 나중에 조심할 점
앞으로 비슷한 요청이 오면 아래를 같이 체크하면 좋습니다.

- 화면에 필드를 추가했는가?
- 수정 모드 초기값도 채워지는가?
- 서버 저장 로직도 그 필드를 받는가?
- 저장 정리 코드가 새 필드를 지우지 않는가?
- 목록 외 상세 화면도 같이 맞춰야 하는가?

이 체크리스트를 습관처럼 쓰면, 작은 기능 추가에서도 실수가 크게 줄어듭니다.

## 다음에 확장한다면
지금 구조를 유지하면서 확장할 수 있는 현실적인 방향은 이 정도입니다.

- URL 유효성 검사 추가
- `URL`과 `URL(Admin)` 복사 버튼 추가
- 관리자 URL은 권한 있는 사용자만 보이게 처리
- 프로젝트 상세 화면에 "서비스 열기 / 관리자 열기" 버튼 분리

중요한 건, 이런 확장은 지금의 단순한 구조를 깨지 않고 위에 얹어야 한다는 점입니다.

## 마지막 요약
이번 프로젝트는 "많은 기능"보다 "정확한 정보 정리"에 강한 앱입니다.

이번 변경도 같은 철학입니다.

- 서비스 주소는 서비스 주소대로
- 관리자 주소는 관리자 주소대로
- 저장과 화면을 모두 같은 구조로 맞추기

겉보기엔 작은 컬럼 하나 추가처럼 보이지만,  
실제로는 데이터 설계와 화면 정합성을 같이 챙긴 실무형 수정입니다.
