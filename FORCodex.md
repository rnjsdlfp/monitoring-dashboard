# FORCodex

## 이 프로젝트를 한 문장으로 말하면
이 프로젝트는 "여러 프로젝트를 한곳에 정리해서 보는 운영 노트"입니다.  
예전에는 각 프로젝트의 서버 상태까지 직접 체크하려고 했지만, 지금은 그 복잡함을 걷어내고 `프로젝트 정보 관리`에 집중한 가벼운 대시보드가 되었습니다.

조금 비유하면 이렇습니다.

- 예전 버전: 경비실 + CCTV + 경보 시스템
- 지금 버전: 잘 정리된 운영 대장 + 빠른 연락처 카드

후자가 덜 화려해 보여도, 실제로는 더 안정적이고 유지보수하기 쉽습니다. 이번 변경의 핵심은 바로 그 점입니다.

## 지금 이 프로젝트가 실제로 하는 일
사용자는 프로젝트를 등록하고, 아래 같은 정보를 한 화면에서 봅니다.

- 프로젝트 이름
- 설명
- 출력 형식
- 서버 위치
- URL
- 기술 스택

그리고 필요하면 수정하거나 삭제할 수 있습니다.  
즉, 이 앱은 "상태 감시기"가 아니라 "프로젝트 운영 정보판" 역할을 합니다.

## 기술 구조를 쉬운 말로 풀면

### 1. `client/` 는 화면입니다
사용자가 직접 보는 웹 화면입니다.

- `ProjectsPage.jsx`
  프로젝트 목록 테이블을 보여줍니다.
- `AddEditModal.jsx`
  프로젝트를 추가하거나 수정하는 입력창입니다.
- `ProjectRow.jsx`
  테이블 한 줄을 그립니다.
- `ProjectAccordion.jsx`
  행을 펼쳤을 때 보이는 상세 카드입니다.
- `SettingsPage.jsx`
  Telegram 연결 정보만 관리합니다.

쉽게 말하면 `client/` 는 손님이 앉는 매장 홀입니다.

### 2. `server/` 는 창고이자 계산대입니다
브라우저 뒤에서 데이터를 받고 저장하는 곳입니다.

- `app.js`
  서버 시작점입니다.
- `routes/projects.js`
  프로젝트 목록 조회, 추가, 수정, 삭제를 처리합니다.
- `routes/settings.js`
  Telegram 설정 조회/저장을 처리합니다.
- `lib/dataStore.js`
  JSON 파일을 읽고 쓰는 공용 저장 도우미입니다.
- `services/telegramService.js`
  Telegram 테스트 메시지를 보내는 기능을 담고 있습니다.

쉽게 말하면 `server/` 는 "입력받은 내용을 정리해서 서랍에 넣는 사무실"입니다.

### 3. 데이터는 DB가 아니라 JSON 파일에 저장합니다
`JSON`  
텍스트 형태로 데이터를 저장하는 파일 형식입니다. 엑셀보다 개발자 친화적인 메모장 표라고 생각하면 됩니다.

이 프로젝트는 복잡한 데이터베이스 대신 JSON 파일을 씁니다.  
이 결정은 "지금 필요한 수준에 맞는 가장 단순한 구조"를 택한 것입니다.

## 데이터가 흐르는 방식

### 프로젝트 추가
1. 사용자가 화면에서 프로젝트 정보를 입력합니다.
2. 브라우저가 `/api/projects` 로 요청을 보냅니다.
3. 서버가 그 데이터를 받아 `projects.json` 에 저장합니다.
4. 화면은 다시 목록을 불러와 바로 보여줍니다.

### 프로젝트 수정
1. 사용자가 기존 프로젝트를 수정합니다.
2. 브라우저가 `/api/projects/:id` 로 수정 요청을 보냅니다.
3. 서버가 같은 프로젝트를 찾아 내용을 덮어씁니다.
4. 화면은 최신 내용으로 다시 그려집니다.

### 프로젝트 삭제
1. 사용자가 삭제 버튼을 누릅니다.
2. 브라우저가 삭제 요청을 보냅니다.
3. 서버가 해당 프로젝트를 제거하고 번호를 다시 정리합니다.
4. 화면에서 즉시 목록이 갱신됩니다.

복잡한 비동기 배치 작업이나 외부 상태 체크는 더 이상 없습니다.  
그래서 흐름을 따라가기가 훨씬 쉽습니다.

## 왜 이렇게 단순하게 바꿨나
이번 프로젝트에서 중요한 교훈이 있습니다.

처음엔 "서버 상태까지 자동으로 체크하면 더 좋아 보이겠지?"라는 생각이 들어가기 쉽습니다.  
하지만 실제로는 다음 문제가 같이 따라옵니다.

- 외부 프로젝트 URL이 죽어 있으면 대시보드도 오류 처리를 해야 함
- health endpoint 형식이 프로젝트마다 다를 수 있음
- schedule heartbeat 방식은 각 프로젝트에 추가 작업이 필요함
- 상태 판단 기준이 애매하면 오탐이 생김
- Telegram 알림까지 붙으면 조용히 있어야 할 앱이 시끄러워짐

즉, 본래 하려던 일은 "프로젝트를 정리해서 보기"였는데, 점점 "모니터링 플랫폼"을 만드는 방향으로 커지고 있었습니다.

좋은 엔지니어는 여기서 한 걸음 물러납니다.

"이 기능이 멋져 보이느냐?"보다  
"이 기능이 지금 제품 목적에 맞느냐?"를 먼저 묻습니다.

이번 변경은 바로 그 판단의 결과입니다.

## 이번에 제거한 것들
다음 기능은 이번 정리에서 제거되었습니다.

- 프로젝트 URL로 직접 요청을 보내는 서버 상태 체크
- `healthEndpoint` 를 호출하는 DB 상태 체크
- heartbeat 기반 schedule 상태 체크
- 수동 모니터링 실행 API
- Projects 화면의 `Server / Schedule / DB / Latest Update` 컬럼
- 상태 입력용 폼 항목들
- 상태 점검용 백엔드 파일들

이렇게 하면서 코드가 훨씬 짧아졌고, 책임도 분명해졌습니다.

## 지금 코드베이스를 읽는 방법
비개발자 입장에서 코드를 볼 때는 아래 순서가 가장 쉽습니다.

### 1. 화면부터 본다
먼저 `client/src/pages/ProjectsPage.jsx` 를 보면 됩니다.

여기서:
- 어떤 표를 보여주는지
- 어떤 버튼이 있는지
- 어떤 컴포넌트를 조합하는지

를 볼 수 있습니다.

### 2. 입력창을 본다
그 다음 `client/src/components/AddEditModal.jsx` 를 보면,
"프로젝트 한 개는 어떤 정보로 이루어져 있지?"가 바로 보입니다.

이 파일은 데이터 구조를 이해하는 가장 쉬운 입구입니다.

### 3. 서버 저장 방식을 본다
그 다음 `server/routes/projects.js` 와 `server/lib/dataStore.js` 를 보면,
"저 정보가 어디에 저장되고 어떻게 다시 읽히는지"를 이해할 수 있습니다.

이 세 단계를 보면 이 프로젝트의 뼈대가 거의 다 보입니다.

## 이번 작업에서 배울 수 있는 실전 교훈

### 1. 단순함은 타협이 아니라 전략입니다
`아키텍처(Architecture)`  
프로그램을 어떤 구조로 나누고 연결할지 정하는 큰 설계입니다.

좋은 아키텍처는 무조건 큰 구조가 아닙니다.  
지금 문제를 가장 적은 복잡도로 풀어내는 구조가 좋은 구조입니다.

### 2. 저장 모델은 화면과 맞아야 합니다
폼에서는 안 쓰는데 서버에는 남아 있는 필드가 많아지면, 언젠가 반드시 혼란이 생깁니다.  
이번에는 프로젝트 데이터 구조 자체를 줄여서 그 문제를 예방했습니다.

### 3. 고아 코드는 빨리 치우는 편이 낫습니다
`고아 코드(Orphan code)`  
지금은 아무도 사용하지 않는데 파일이나 함수만 남아 있는 코드입니다.

이런 코드는 미래의 나를 헷갈리게 합니다.  
그래서 이번에는 상태 점검 파일과 신호등 컴포넌트를 함께 제거했습니다.

### 4. "할 수 있음"과 "해야 함"은 다릅니다
프로젝트 URL을 주기적으로 호출하는 기능은 만들 수 있습니다.  
하지만 지금 제품의 핵심 가치가 그 기능이 아니라면, 오히려 빼는 것이 더 좋은 결정일 수 있습니다.

## 실제로 부딪힌 문제와 해결

### 문제 1. 상태 기능이 화면, 서버, 저장 구조에 넓게 퍼져 있었습니다
처음에는 "테이블 컬럼만 지우면 되겠지"처럼 보일 수 있습니다.  
하지만 실제로는:

- 화면 컬럼
- 프로젝트 입력 폼
- 서버 라우트
- JSON 저장 구조
- 보조 서비스 파일

까지 연결되어 있었습니다.

#### 어떻게 풀었나
겉 UI만 숨기지 않고, 데이터 구조부터 같이 줄였습니다.  
그래야 나중에 "숨겨져 있지만 살아 있는 오래된 기능"이 남지 않습니다.

### 문제 2. 오래된 문자열과 화면 조각이 엉켜 있었습니다
기존 파일 일부에는 상태 기능을 전제로 한 문구와 구조가 섞여 있었습니다.

#### 어떻게 풀었나
조금씩 뜯어고치기보다, 핵심 화면 파일은 통째로 다시 간단하게 작성했습니다.  
이 방법이 오히려 더 안전했습니다.

### 문제 3. 문서가 예전 아키텍처를 설명하고 있었습니다
코드만 바꾸면 끝이 아닙니다.  
비개발자는 문서를 보고 시스템을 이해하므로, 문서가 틀리면 오히려 더 위험합니다.

#### 어떻게 풀었나
`FORCodex.md` 와 `PROGRESS.MD` 를 현재 코드 기준으로 다시 정리했습니다.

## 나중에 확장하고 싶다면
지금은 단순한 구조가 맞지만, 나중에 아래 같은 확장은 가능합니다.

### 1. 검색/필터 추가
프로젝트 수가 늘어나면 이름, 기술 스택, 서버 위치 기준 필터가 유용합니다.

### 2. 정식 DB 도입
프로젝트 수가 많아지고 변경 이력 관리가 중요해지면 SQLite나 PostgreSQL로 넘어갈 수 있습니다.

### 3. 링크 품질 개선
URL 형식 검증, 새 창 열기 정책, 복사 버튼 같은 편의 기능을 넣을 수 있습니다.

단, 좋은 확장은 "지금 있는 단순함"을 깨지 않는 방식이어야 합니다.

## 좋은 엔지니어처럼 생각하는 법
이 프로젝트에서 꼭 기억하면 좋은 질문은 이것입니다.

- 이 기능은 실제 목적을 더 잘 달성하게 해주는가?
- 지금 사용자에게 필요한가?
- 코드와 데이터 구조를 같이 단순하게 만들 수 있는가?
- 나중에 읽는 사람이 덜 헷갈리게 만들었는가?

이 질문을 반복하면, 기능을 더 많이 넣는 사람보다 더 좋은 시스템을 만드는 사람에 가까워집니다.

## 마지막 비유
이 프로젝트는 지금 "자동 감시 타워"가 아니라 "정리 잘 된 운영 보드"입니다.

보드가 해야 할 일은 명확합니다.

- 필요한 정보를 보여주고
- 쉽게 수정하게 하고
- 복잡한 자동 판단은 함부로 끌어오지 않는 것

겉보기 화려함은 줄었지만, 제품의 중심은 더 또렷해졌습니다.  
그리고 대부분의 실무 프로젝트는 이런 방향의 정리가 오히려 오래 갑니다.
