# FORCodex

## 이 프로젝트를 한 문장으로 말하면
이 프로젝트는 여러 앱과 서버의 상태를 한 화면에서 보고, 문제가 생기면 Telegram으로 알려주고, 하루 요약까지 보내주는 "작은 관제실"입니다.

## 먼저, 아주 쉬운 그림으로 이해하기
- `프론트엔드(Frontend)`: 사용자가 보는 화면입니다. 식당으로 치면 손님이 보는 홀입니다.
- `백엔드(Backend)`: 화면 뒤에서 데이터 저장, 상태 점검, 알림 발송을 처리하는 부분입니다. 식당으로 치면 주방과 관리실입니다.
- `라우터(Router)`: 어떤 주소로 요청이 오면 누구에게 일을 맡길지 정하는 분배기입니다.
- `서비스(Service)`: 실제 중요한 업무를 처리하는 실무 코드입니다.
- `JSON`: 데이터베이스 대신 쓰는 텍스트 파일입니다. 작은 가게의 수기 장부 같은 역할입니다.

이 대시보드는 크게 두 팀이 같이 일합니다.

1. 화면 팀
사용자가 프로젝트를 추가하고, 상태를 보고, 설정을 바꾸는 일을 맡습니다.

2. 운영 팀
서버 상태를 체크하고, 스케줄 신호를 받고, Telegram 메시지를 보내는 일을 맡습니다.

## 왜 이렇게 만들었나
처음부터 가장 단순하고 안정적인 구조를 고른 것이 핵심입니다.

- `React + Vite`
화면 개발 속도가 빠릅니다. 복잡한 설정 없이 바로 UI를 만들기 좋습니다.

- `Tailwind CSS`
작은 스타일 조각을 바로 붙여가며 화면을 만들 수 있어서, 초기 구현 속도가 빠릅니다.

- `Node.js + Express`
API 서버를 가볍게 만들기 좋습니다. 구조가 단순해서 나중에 고치기도 쉽습니다.

- `JSON 파일 저장`
지금 단계에서는 DB를 붙이는 것보다 훨씬 단순합니다. 운영 복잡도를 줄여 줍니다.

- `Telegram`
별도 앱을 만들지 않아도 즉시 알림을 받을 수 있습니다.

좋은 엔지니어는 "멋있어 보이는 구조"보다 "지금 문제를 가장 적은 비용으로 안정적으로 푸는 구조"를 먼저 고릅니다. 이번 프로젝트가 딱 그 원칙을 따릅니다.

## 전체 구조

### 1. 화면 구조
`client/` 아래에 화면 코드가 있습니다.

- `src/App.jsx`
전체 레이아웃의 출발점입니다. 왼쪽 사이드바와 오른쪽 본문 영역을 나눕니다.

- `src/context/ThemeContext.jsx`
다크 모드와 라이트 모드를 전역으로 관리합니다.

- `src/components/Sidebar.jsx`
프로젝트 목록, Settings 이동, 테마 토글을 보여줍니다.

- `src/pages/ProjectsPage.jsx`
프로젝트 표, 아코디언 상세 보기, 추가/수정/삭제 흐름을 담당합니다.

- `src/pages/SettingsPage.jsx`
Telegram 설정, 즉시 알림 토글, Daily Report 토글, 수동 모니터링 실행, heartbeat API 안내를 담당합니다.

- `src/components/AddEditModal.jsx`
프로젝트를 새로 만들거나 수정할 때 뜨는 팝업입니다.

- `src/components/ProjectRow.jsx`
표 한 줄의 상태와 버튼을 그립니다.

- `src/components/ProjectAccordion.jsx`
행을 눌렀을 때 아래로 펼쳐지는 상세 정보 영역입니다.

- `src/components/StatusLight.jsx`
초록/노랑/빨강/회색 상태등을 보여줍니다.

### 2. 서버 구조
`server/` 아래에 운영 로직이 있습니다.

- `app.js`
Express 서버 시작점입니다. API를 연결하고, 모니터링 서비스와 Daily Report 서비스를 함께 시작합니다.

- `routes/projects.js`
프로젝트 생성/조회/수정/삭제 API입니다.

- `routes/heartbeat.js`
각 프로젝트의 스케줄 작업이 살아 있다는 신호를 받는 API입니다.

- `routes/settings.js`
Telegram 설정과 리포팅 설정을 읽고 저장하는 API입니다.

- `services/monitorService.js`
모든 프로젝트의 서버/스케줄/DB 상태를 점검합니다.

- `services/telegramService.js`
Telegram 메시지를 보내는 공통 모듈입니다.

- `services/dailyReportService.js`
매일 정해진 시간에 요약 메시지를 보냅니다.

- `lib/dataStore.js`
JSON 파일 읽기/쓰기 공통 처리입니다.

- `lib/time.js`
KST 시간 포맷을 맞추는 도우미입니다.

## 데이터는 어떻게 흘러가나
이 프로젝트의 흐름은 생각보다 단순합니다.

### 흐름 A: 사용자가 프로젝트를 추가한다
1. 화면의 모달에서 정보를 입력합니다.
2. 프론트엔드가 `/api/projects`로 POST 요청을 보냅니다.
3. 백엔드가 `projects.json`에 새 항목을 저장합니다.
4. 화면이 다시 목록을 불러와 표와 사이드바를 갱신합니다.

### 흐름 B: 서버가 자동 상태를 체크한다
1. `monitorService`가 5분마다 실행됩니다.
2. 각 프로젝트 URL에 요청을 보내 Server 상태를 확인합니다.
3. health endpoint가 있으면 DB 상태도 확인합니다.
4. heartbeat 기록을 보고 Schedule 상태를 판단합니다.
5. 결과를 `projects.json`에 저장합니다.
6. 상태가 위험하게 바뀌면 `eventLog.json`에 기록하고 Telegram 알림을 보냅니다.

### 흐름 C: 스케줄 작업이 heartbeat를 보낸다
1. 외부 프로젝트의 cron job이 `/api/heartbeat`로 POST 요청을 보냅니다.
2. 서버는 `heartbeats.json`에 마지막 수신 시각을 저장합니다.
3. 해당 프로젝트의 Schedule 상태를 업데이트합니다.
4. 성공 heartbeat는 `eventLog.json`에도 남겨 Daily Report에 반영됩니다.

### 흐름 D: Daily Report가 만들어진다
1. 매분 현재 KST 시간을 확인합니다.
2. 설정된 시각이 되면 지난 24시간 이벤트를 모읍니다.
3. 프로젝트별 오류/복구/스케줄 성공 횟수를 계산합니다.
4. Telegram으로 요약 메시지를 보냅니다.

## 코드에서 중요한 판단들

### 1. 왜 DB 대신 JSON인가
지금 목표는 "빠르게 안정적인 대시보드 만들기"입니다. DB를 붙이면 설정, 마이그레이션, 백업, 배포 포인트가 늘어납니다. 지금은 그 복잡도를 살 이유가 적습니다.

배워둘 점:
좋은 엔지니어는 "미래에 언젠가 필요할지도 모를 확장성" 때문에 현재를 복잡하게 만들지 않습니다.

### 2. 왜 공통 저장 유틸리티를 만들었나
같은 JSON 읽기/쓰기를 여러 파일에서 반복하면, 작은 실수 하나가 여러 군데로 퍼집니다. 그래서 `dataStore.js`로 공통화했습니다.

배워둘 점:
반복이 두세 군데 이상 생기면, 그때는 묶는 것이 오히려 더 단순합니다.

### 3. 왜 상태 변화에만 즉시 알림을 보내나
계속 빨간 상태인 서버에 5분마다 계속 알림을 보내면 사람은 금방 무감각해집니다. 그래서 "나빠졌을 때 한 번", "복구됐을 때 한 번"만 보내도록 했습니다.

배워둘 점:
모니터링은 "많이 보내는 것"이 아니라 "의미 있게 보내는 것"이 중요합니다.

## 구현하면서 실제로 부딪힌 문제들

### 문제 1. 수동 모니터링 API 경로가 처음엔 잘못 붙었다
처음 구현에서는 `/api/monitor/run`이 아니라 라우터 구조상 `/api/projects/monitor/run`처럼 붙을 위험이 있었습니다.

어떻게 고쳤나:
`app.js`에서 `/api/monitor/run`을 명시적으로 연결해서 `PLAN.md`와 실제 경로를 맞췄습니다.

배울 점:
문서, 라우터 마운트 경로, 실제 호출 URL은 항상 같이 봐야 합니다.

### 문제 2. Daily Report 전송 실패 시 이벤트 로그를 지우면 안 됐다
처음 버전에서는 Telegram 전송이 실패해도 이벤트 정리가 진행될 수 있는 흐름이 있었습니다.

어떻게 고쳤나:
메시지 전송이 성공했을 때만 로그를 정리하고 `lastSentDate`를 갱신하도록 바꿨습니다.

배울 점:
알림 시스템은 "전송 성공"과 "처리 완료"를 같은 것으로 보면 안 됩니다.

### 문제 3. Windows 환경에서 큰 패치가 한 번에 안 들어갔다
길이가 긴 한 번짜리 파일 수정은 Windows 명령 길이 제한에 걸렸습니다.

어떻게 고쳤나:
파일을 여러 번에 나눠 적용했습니다.

배울 점:
기술 문제는 코드가 아니라 도구 제약일 때도 많습니다. 원인을 먼저 좁히는 습관이 중요합니다.

### 문제 4. 테스트 데이터가 실제 데이터처럼 남을 수 있었다
브라우저 자동화로 프로젝트를 추가하고 토글을 켠 뒤 그대로 두면, 나중에 사용자가 "이게 원래 데이터인가?" 헷갈릴 수 있습니다.

어떻게 고쳤나:
테스트 후 `server/data/*.json`을 기본 상태로 다시 정리했습니다.

배울 점:
좋은 작업 마무리는 "기능 구현"에서 끝나지 않고 "작업 흔적 정리"까지 포함합니다.

## 좋은 엔지니어처럼 이 프로젝트 읽는 법

### 1. 먼저 큰 흐름을 본다
처음부터 한 줄 한 줄 읽지 말고, "이 앱은 어떤 화면이 있고, 어떤 API가 있고, 어떤 파일에 저장하는가?"를 먼저 잡으세요.

### 2. 그다음 데이터 흐름을 본다
프로젝트 추가 버튼을 누르면 어디로 가는지, heartbeat가 오면 어떤 파일이 바뀌는지 추적하면 구조가 훨씬 빨리 보입니다.

### 3. 마지막으로 예외 상황을 본다
토큰이 비어 있을 때, 서버가 죽었을 때, Daily Report가 실패했을 때 어떻게 되는지 보면 코드 품질이 보입니다.

## 나중에 확장하고 싶다면

### 1. DB로 바꾸기
프로젝트 수가 많아지고 이력 관리가 중요해지면 SQLite나 PostgreSQL로 옮길 수 있습니다.

### 2. 인증 추가
지금은 내부용 도구라 로그인 없이도 가능하지만, 외부 노출 시에는 인증이 필요합니다.

### 3. 프로젝트별 세부 로그 화면
지금은 표 중심이므로, 프로젝트별 상태 이력 차트나 이벤트 타임라인을 붙일 수 있습니다.

### 4. Telegram 외 채널 추가
이메일, Discord, Slack 같은 채널도 같은 서비스 패턴으로 붙일 수 있습니다.

## 이 프로젝트에서 꼭 기억할 교훈
- 단순한 설계는 게으른 설계가 아니라, 관리 비용이 낮은 좋은 설계일 수 있습니다.
- 모니터링 시스템은 "상태 수집", "상태 저장", "상태 변화 알림"을 분리해서 생각하면 훨씬 명확해집니다.
- 비개발자와 같이 일할수록 문서와 흐름 설명이 중요합니다. 코드를 짜는 것만으로는 협업이 끝나지 않습니다.
- 기능 구현 뒤에는 반드시 검증과 정리가 와야 합니다.

## 마지막으로, 이 프로젝트를 기억하는 비유
이 대시보드는 건물 경비실과 비슷합니다.

- 표는 CCTV 모니터입니다.
- heartbeat는 각 방에서 오는 "이상 없음" 무전입니다.
- Telegram 알림은 경비실에서 휴대폰으로 보내는 긴급 문자입니다.
- Daily Report는 아침 인수인계 노트입니다.

즉, 이 프로젝트의 핵심은 화려한 UI가 아니라 "무슨 일이 벌어지고 있는지 놓치지 않게 만드는 것"입니다. 좋은 운영 도구는 원래 이런 성격을 가집니다.
