# FORCodex

## 이 프로젝트를 한 문장으로 말하면
이 프로젝트는 "여러 운영 프로젝트를 한 화면에서 저장하고, 다시 찾고, 수정하고, 순서를 정리하는 개인용 대시보드"입니다.

예전에는 상태 감시 성격이 더 강했지만, 지금의 방향은 훨씬 선명합니다.  
"뭔가를 계속 감시하는 도구"라기보다 "프로젝트 정보를 깔끔하게 보관하는 선반"에 가깝습니다.

비유하면 이렇습니다.

- 예전 느낌: 경보가 울릴 수도 있는 관제실
- 지금 느낌: 자주 꺼내보는 프로젝트 카드 정리함

이 변화는 작아 보이지만 꽤 중요합니다. 좋은 제품은 기능 수보다 목적이 또렷한지가 더 중요하기 때문입니다.

## 이번 변경의 핵심
이번 작업에서는 저장된 `Projects` 화면에 두 가지를 완성했습니다.

- `CRUD`
  `CRUD`는 Create, Read, Update, Delete의 줄임말입니다.  
  즉, 만들기, 보기, 수정하기, 삭제하기입니다.
- `drag & drop`
  `drag & drop`은 마우스로 항목을 끌어서 다른 위치에 놓는 방식입니다.  
  목록 순서를 직관적으로 바꿀 때 많이 씁니다.

정리하면 이제 이 화면은 단순히 "프로젝트를 보여주는 표"가 아니라, 실제로 관리하는 작업대가 되었습니다.

## 지금 저장되는 정보
각 프로젝트는 아래 정보를 가집니다.

- `name`: 프로젝트 이름
- `description`: 프로젝트 설명
- `outputFormat`: 결과물이 어떤 형태인지
- `serverLocation`: 서버가 어디 있는지
- `url`: 서비스 주소
- `adminUrl`: 관리자 주소
- `techStack`: 사용 기술 목록
- `no`: 화면 정렬 순서
- `id`: 각 프로젝트를 구분하는 고유 ID
- `createdAt`: 처음 저장한 시각

여기서 중요한 포인트는 `no`입니다.  
이 값이 단순 번호처럼 보여도, 실제로는 "화면에서 어떤 순서로 보일지"를 결정하는 정렬 기준입니다.

## 코드 구조를 쉬운 말로 설명하면

### `client/`
사용자가 직접 보는 화면입니다. React로 만들었습니다.

- `client/src/pages/ProjectsPage.jsx`
  프로젝트 목록 화면의 중심입니다.
  목록 불러오기, 추가/수정 모달 열기, 삭제, 드래그 정렬 저장까지 여기서 조율합니다.
- `client/src/components/ProjectRow.jsx`
  표의 한 줄을 그립니다.
  이번에 드래그 핸들, 드롭 위치 표시선, 드래그 중 투명도 처리가 들어갔습니다.
- `client/src/components/ProjectAccordion.jsx`
  행을 눌렀을 때 아래로 펼쳐지는 상세 정보 카드입니다.
- `client/src/components/AddEditModal.jsx`
  새 프로젝트를 만들거나 기존 프로젝트를 수정하는 입력 창입니다.

### `server/`
저장과 읽기를 담당하는 뒤쪽 로직입니다. Express 서버입니다.

- `server/app.js`
  서버 시작점입니다.
- `server/routes/projects.js`
  프로젝트 API가 들어 있습니다.
  조회, 생성, 수정, 삭제에 더해 이번에 순서 재배치 API가 추가되었습니다.
- `server/lib/dataStore.js`
  실제 JSON 파일을 읽고 쓰는 저장 담당입니다.

## 데이터가 어떻게 흐르는가

### 프로젝트 추가
1. 사용자가 `Add Project`를 누릅니다.
2. `AddEditModal.jsx`에서 값을 입력합니다.
3. `ProjectsPage.jsx`가 `POST /api/projects` 요청을 보냅니다.
4. `server/routes/projects.js`가 새 프로젝트를 만들고 저장합니다.
5. `dataStore.js`가 JSON 파일에 기록합니다.
6. 화면이 목록을 다시 읽어와 즉시 반영합니다.

### 프로젝트 수정
1. 사용자가 행의 수정 버튼을 누릅니다.
2. 기존 값이 모달에 채워집니다.
3. `PUT /api/projects/:id` 요청이 서버로 갑니다.
4. 서버가 같은 ID의 프로젝트를 찾아 값을 덮어씁니다.
5. 저장 후 목록을 다시 읽어 화면을 갱신합니다.

### 프로젝트 삭제
1. 사용자가 삭제 버튼을 누릅니다.
2. 화면에서 한 번 더 확인합니다.
3. `DELETE /api/projects/:id` 요청이 서버로 갑니다.
4. 서버가 해당 프로젝트를 제거합니다.
5. 남은 프로젝트들의 `no`를 다시 매겨 순서를 정리합니다.

### 프로젝트 순서 바꾸기
1. 사용자가 행의 드래그 핸들을 잡고 끕니다.
2. 화면은 먼저 브라우저 안에서 순서를 바꿔 보여줍니다.
3. 이어서 `PUT /api/projects/reorder` 요청으로 새 순서를 서버에 보냅니다.
4. 서버는 받은 ID 배열 순서대로 프로젝트를 다시 정렬합니다.
5. `writeProjects()`가 저장하면서 `no`를 1, 2, 3... 순서로 다시 매깁니다.
6. 새로고침해도 순서가 유지됩니다.

이 흐름이 중요한 이유는 "눈에만 바뀌는 순서"가 아니라 "저장되는 순서"가 되었기 때문입니다.

## 왜 이렇게 구현했는가

### 1. 새 라이브러리를 넣지 않았다
이번 정렬 기능은 별도 드래그 라이브러리 없이 브라우저 기본 기능으로 만들었습니다.

이 선택의 장점:

- 설치가 필요 없습니다.
- 코드가 짧습니다.
- 유지보수 포인트가 줄어듭니다.
- 지금 화면 규모에는 충분합니다.

좋은 엔지니어는 "멋진 도구"보다 "지금 문제에 맞는 가장 작은 도구"를 먼저 봅니다.

### 2. 정렬 전용 API를 따로 만들었다
처음엔 "프런트에서 배열만 바꾸면 끝 아닌가?"라고 생각할 수 있습니다.  
하지만 그건 새로고침하면 사라지는 변화일 뿐입니다.

그래서 서버에 `PUT /api/projects/reorder`를 따로 만들었습니다.

이 선택의 장점:

- 순서가 진짜 저장됩니다.
- 브라우저와 서버가 같은 정렬 기준을 공유합니다.
- 나중에 다른 화면이나 다른 사용자 환경이 생겨도 같은 규칙을 재사용할 수 있습니다.

### 3. `no`를 계속 다시 계산한다
삭제나 재정렬 뒤에 번호를 그대로 두면 `1, 2, 5, 9`처럼 중간이 비는 일이 생길 수 있습니다.

그래서 저장할 때마다 `no`를 다시 정리합니다.

이 방식의 장점:

- 정렬 기준이 단순합니다.
- 사람이 보기에도 이해하기 쉽습니다.
- 버그가 생길 여지가 적습니다.

## 이번 작업에서 부딪힌 실제 포인트

### 1. 드래그는 화면에서만 되면 반쪽짜리다
드래그로 순서가 바뀌는 것처럼 보여도, 서버에 저장하지 않으면 새로고침 순간 원래대로 돌아갑니다.

이번 작업의 본질은 "드래그 UI 추가"가 아니라 "드래그 결과를 서버가 이해하게 만들기"였습니다.

### 2. 드롭 위치는 위/아래를 구분해야 한다
행 위에 놓았는지, 아래에 놓았는지에 따라 결과가 달라집니다.

그래서 마우스 위치가 행의 절반보다 위인지 아래인지 계산해서 `before` 또는 `after`로 처리했습니다.

이건 작은 디테일 같지만, 실제 사용감에 큰 차이를 만듭니다.

### 3. 낙관적 반영과 실제 저장은 다르다
`optimistic update`는 서버 응답을 기다리지 않고 먼저 화면을 바꾸는 방식입니다.  
사용자는 더 빠르게 느끼지만, 저장 실패 시 되돌리는 처리도 필요합니다.

이번 코드도 이 방식을 썼습니다.

- 먼저 화면 순서를 바꿉니다.
- 서버 저장이 실패하면 이전 순서로 복구합니다.

즉, 빠름과 안정성을 같이 잡으려는 선택입니다.

## 배울 점

### 1. 기능은 화면 하나만 고치면 끝나지 않는다
이번 요청은 얼핏 보면 "행을 드래그하게 해주세요"입니다.  
하지만 실제로는 아래가 함께 바뀌어야 했습니다.

- 프런트 상태 관리
- 시각적 피드백
- 서버 API
- 파일 저장 규칙
- 진행 기록 문서
- 인수인계 문서

좋은 엔지니어는 요청 문장을 그대로만 보지 않고, 그 기능이 시스템 어디까지 영향을 주는지 같이 봅니다.

### 2. 단순한 구조는 강하다
이 프로젝트는 데이터베이스 대신 JSON 파일을 씁니다.

큰 서비스라면 부족할 수 있지만, 지금 단계에서는 오히려 장점이 있습니다.

- 디버깅이 쉽습니다.
- 데이터 구조를 눈으로 바로 볼 수 있습니다.
- 비개발자도 시스템을 이해하기 쉽습니다.

단순함은 임시방편이 아니라, 상황에 맞으면 가장 강한 설계가 됩니다.

### 3. 검증은 "돌아가 보인다"로 끝내면 안 된다
이번 작업 뒤에는 최소한 아래를 확인해야 했습니다.

- 프런트 빌드가 되는가
- 서버 파일 문법이 깨지지 않았는가
- 재정렬 API 경로가 기존 `/:id` 경로보다 먼저 선언되어 충돌하지 않는가

특히 마지막은 실무에서 자주 놓치는 포인트입니다.  
정적 경로인 `/reorder`를 매개변수 경로인 `/:id` 뒤에 두면 엉뚱하게 해석될 수 있습니다.

## 앞으로 조심할 점

- 프로젝트가 아주 많아지면 JSON 파일 기반 저장은 한계가 올 수 있습니다.
- 여러 사용자가 동시에 순서를 바꾸면 마지막 저장이 이기는 구조가 됩니다.
- 드래그 앤 드롭은 모바일에서 데스크톱만큼 자연스럽지 않을 수 있습니다.

즉, 지금 구조는 "작고 빠르게 관리하는 내부 도구"에는 매우 좋지만, 규모가 커지면 다음 단계 설계가 필요할 수 있습니다.

## 다음에 확장한다면

- `name` 필수값 검증 추가
- `url`, `adminUrl` 형식 검증 추가
- 드래그 말고도 위/아래 이동 버튼 추가
- 모바일 친화적인 재정렬 UI 추가
- 데이터베이스로 이전

중요한 점은, 이런 확장은 "지금 반드시 필요한가?"를 먼저 따져야 한다는 것입니다.  
좋은 개발은 미래를 상상하는 능력도 중요하지만, 미래를 핑계로 현재를 복잡하게 만들지 않는 절제도 중요합니다.

## 실행과 확인에 쓰는 명령어
아래 명령은 PowerShell 기준입니다.

클라이언트 빌드 확인:

```powershell
cd C:\Users\JIREH\app\project-dashboard\client
npm.cmd run build
```

서버 문법 확인:

```powershell
cd C:\Users\JIREH\app\project-dashboard\server
node --check routes\projects.js
node --check lib\dataStore.js
```

## 마지막 한 줄 요약
이번 변경은 단순히 "끌어다 놓는 UI"를 붙인 작업이 아닙니다.  
저장된 프로젝트 목록을 실제 관리 도구답게 만들기 위해, 화면 조작과 서버 저장 규칙을 같은 방향으로 맞춘 작업입니다.
